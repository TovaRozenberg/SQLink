## פרוייקט ניהול מערכת מכירות פומביות
הפרוייקט המצורף מכיל מערכת ניהול מכירות פומביות,
המערכת מורכבת מצד שרת וצד לקוח

#### צד שרת
בצד זה הכנתי קונטרולרים, העדפתי אותם על פני minimal api מאחר וזה הקוד שאני מכירה, וכן רציתי ככל הניתן לדמות מערכת גדולה, שם מסודר יותר הקונטרולרים
כנ"ל לגבי העדפתי את SQL server על פני SQLligiht, אמנם השני קל ונוח יותר, אך רציתי לדמות מערכת גדולה 
את הDB ממשתי בשיטת code first כי C# יותר קל לי לכתוב מ SQL... ולכן התקנתי את הספריות של SQL הממירות את הEntities לטבלאות SQL
טרנזאקציות מומשו ע"י אותן ספריו (לא זוכרת את השם בע"פ) ובכך הבטחתי את בטיחות הטבלאות ומניעת הבעיה של עדכון מקבילי
כדי לטפל בבעית עדכון בזמן אמת, הוספתי signalR שמתחזק מרכזיה ובה נכנסים כל המשתמשי למערכת, (בצד הלקוח ממשתי את הכניס והיציאה ג"כ),
מתי שהוכנסה הצעת מחיר חדשה, הוא  מעדכן את כל המשתמשים הרשומים אצלו בנתון החדש, ובצד השרת הם מקבלים ומעדכנים את המסכים של הלקוחות בזמן אמת
בנוסף על מנת לעדכן משתמשים פעילים בהורדת והוספת מכירות , השתמשתי ג"כ בדבר זהה
וגם לעדכון המחיר הגבוה ביותר הנוכחי
הSignalR , משתמש (נראה לי) הצינור פתוח בינו לבין הלקוח, ושם הוא שולח לו עדכונים
מכיון שעשיתי מערכת הרשמה והתחברות, השתמשי ב JWT עמ לאבטח את הקריאות API ולמנוע זליגת מידה למשתמשים לא מחוברים
ע"מ לטפל באתגר העיקרי אני מתכננת (עוד לא הספקתי...) להוסיף כעין תור (אני משערת שיש ספריה שמממשת את זה) ומתי שמשתמש מכניס הצעת מחיר הוא נכנס לתור וכל הזמן המערכת המערכת מוציאה מהתור, ומטפלת ומעדכת וכו' כך אם 2 משתמשים נכנסים ביחד, הם יכנסו לתור אחד אחרי השני, ויטופלו אחד אחד 
ע"מ שהמערכת תהיה מסודרת וקלה לשינויים ומעקב הכנתי אותה במבנה של clean API וחלקתי אותה לרשויות המתקשרות עי שכבת ה Core 
עדין לא הוספתי, ואני צריכה להסיף ולדציות על ה DTO של יצירה כדי למנוע מצב בו משתמש מצליח להכניס רטים שגויים כמו הצעת מחי נמוכה מהמחיר הנוכחי, שלילית וכו
בואלידציות הללו, לא אוכל להשתמש באנוטציה (נראה לי כך קוראים לזה: []) מעל מאפיינים, אני אצטרך להתשתמש בואלידציות מתקדמות יותר (שוב, יש סיפריה שעוזרת לזה, לא זוכרת את שמה, אם זה חשוב, אני אבדק....)
